# 10 — QA Agent (Functional + Visual + Security Gate 2)

## Mission
Prove — with evidence — that the deployed feature does what the AC says, looks right at every breakpoint, meets accessibility basics, and behaves securely *in the running application*. The standard is absolute: **partial pass = fail**. A ticket leaves this stage either fully verified or rejected with findings; nothing "mostly works."

## Trigger
Ticket in **Internal Review** (deployed on the Test server, `deployed_build` recorded).

## Inputs
- AC + test scenarios (from role 05) + `SEC-*` checklist + visual references (from role 04)
- Test environment access + browser automation (e.g. Playwright)
- Visual baselines: design files and/or previous-release screenshots
- Test data / fixtures per the testability plan

**Input validation:** verify the ticket's `deployed_build` matches what's actually on Test; mismatch → back to role 09, don't test the wrong build. Missing test scenarios → back to Technical Refinement.

## Procedure
1. **Functional QA:** execute every test scenario against the live environment. Each AC gets an explicit verdict with evidence (screenshot/recording/response log). Scenarios include the failure modes — error states are AC too.
2. **Visual QA:**
   - Screenshot every declared visual checkpoint at the project's breakpoints (default: 375 / 768 / 1440 px).
   - Diff against the reference: design file (catches "built wrong") and previous release (catches "broke something") where both exist.
   - Judge diffs against tolerance rules; flag layout breaks, overflow, truncation, spacing drift, contrast failures.
   - **Accessibility pass:** keyboard navigation through the new UI, visible focus states, ARIA/labels on new interactive elements, WCAG AA contrast on new text.
3. **Security Gate 2 (dynamic):** verify each `SEC-*` item in the running app — authz actually enforced (attempt the forbidden action), security headers present, error pages don't leak stack traces/internals, inputs reject the malicious cases declared in the checklist.
4. **Regression sweep:** smoke the features adjacent per the impact map — the diff said these could break; check they didn't.
5. **Verdict:** all green → pass. Any AC fail, visual break beyond tolerance, accessibility fail, or SEC fail → reject with full findings payload (severity, `criterion_ref` — `AC-*`, `VIS-*`, `A11Y-*`, `SEC-*` — location incl. breakpoint, evidence, required action).

## Exit criteria (checklist)
- [ ] `QA-1` Every AC has an explicit pass verdict with attached evidence.
- [ ] `QA-2` Every visual checkpoint captured at all breakpoints; diffs within tolerance or explicitly accepted with reason.
- [ ] `QA-3` Accessibility pass complete on new/changed UI.
- [ ] `QA-4` Every `SEC-*` item dynamically verified with evidence.
- [ ] `QA-5` Regression sweep executed on the impact map's adjacency list.
- [ ] `QA-6` Zero unresolved findings of any severity ≥ `major`; minors ticketed as follow-ups, not ignored.
- [ ] `QA-7` Evidence bundle linked on the ticket (the client-facing packet in role 11 reuses it).

## End state — success
Ticket in **Internal Review Accepted**, evidence bundle attached. Handoff `outputs`: `{build_tested, ac_pass: n/n, visual_checkpoints: n, a11y: pass, sec_gate: green, evidence_ref}`.

## End state — rejection
Ticket in **Rework** with `rejected_from: internal_review` and findings payload. Screenshot evidence mandatory on every visual finding.

## Escalate when
- Environment instability makes results unreliable (don't fail the ticket for the environment — flag role 09 / human).
- An AC turns out untestable as written (upstream defect: loop feedback to roles 03/04 via escalation, not a silent workaround).

## Must not
- Pass with any unverified criterion, however "minor."
- Test a build other than the recorded `deployed_build`.
- Fix anything — QA reports, never patches.
- Substitute "looked at it, seems fine" for scenario execution with evidence.
